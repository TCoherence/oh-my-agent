# v1.0 计划：稳定的 Discord-First 自托管 Agent Runtime

## Summary

`1.0` 的定义不是"能力更强"，而是"产品契约稳定"。

基于当前 repo 现状，`1.0` 应明确为：

- **single-user / owner-operated**
- **self-hosted first**
- **generic runtime first**
- **reliability first**
- **Discord-only 作为 1.0 平台范围**
- **平台抽象要做到 future-ready，但第二 IM 平台不进入 1.0 gate**

版本路径重定为：

- `0.8` = **1.0 hardening**
- `0.9` = **1.0 RC / contract freeze**
- `1.0` = **稳定发布**
- Slack / Feishu / WeChat、semantic retrieval、hybrid autonomy、guest mode、event-driven triggers、remote object storage 全部移到 **post-1.0 / 1.x**

## 1.0 Product Contract

`1.0` 的承诺是：一个 owner 能在本地或 Docker 中部署系统，接入 Discord，使用 Claude/Gemini/Codex，运行 skills 和长任务，完成审批，重启后恢复，并在不读源码的情况下定位问题。

`1.0` 范围内必须稳定支持：

- 部署：
  - 本地 `venv`
  - Docker
  - `docker-compose` 作为默认运维路径
- Agents：
  - Claude CLI
  - Gemini CLI
  - Codex CLI
- Platform：
  - Discord
- Execution modes：
  - direct chat
  - explicit skill invocation
  - `artifact`
  - `repo_change`
  - `skill_change`
  - file-driven automations
- HITL：
  - structured single-choice only
  - owner-gated approvals / confirmations / checkpoints
- Delivery：
  - inline summary
  - attachment when possible
  - local path fallback
- Persistence / recovery：
  - sessions
  - runtime tasks
  - HITL prompts
  - auth waits
  - automation runtime state
- Operator surfaces：
  - task status / logs
  - automation status / reload / enable / disable
  - auth status / login / clear
  - `/doctor`

`1.0` 明确不做：

- multi-user / team tenancy
- guest mode
- 第二 IM 平台作为发布门槛
- event-driven triggers
- free-text HITL
- remote object storage delivery
- semantic retrieval / vector memory
- hybrid autonomy
- 把 domain-specific skills 当作核心发布标准

## v0.8 — 1.0 Hardening

### 1. Platform abstraction

目标不是"为了 1.0 就上第二平台"，而是把当前 Discord 适配器中的业务逻辑拆出来，降低维护风险，并为后续 Feishu/Slack 留出口。

架构选择：

- **service-layer architecture**
- 平台适配器只负责：
  - 输入解析
  - 输出渲染
  - 平台 SDK 调用
- 共享 service 层负责：
  - 业务逻辑
  - 状态变迁
  - 结果对象

拆分顺序锁定为：

1. **task control service**
   - 当前价值最高，命令最多，状态逻辑最重
2. **ask service**
   - 核心入口路径，必须尽早脱离平台耦合
3. **doctor service**
   - 逻辑小，适合早收口
4. **automation service**
   - 逻辑中等，但 operator 面重要
5. **auth service**
   - 可后置
6. **memory/operator service**
   - `memories/forget/promote` 可最后再拆

要求：

- 不要求一次把所有命令都抽完才算完成
- `0.8` 的完成标准是：
  - 最重的业务逻辑已不再堆在平台 adapter 中
  - 后续新增第二平台时不需要复制 Discord 的业务分支

### 2. BaseChannel contract review

平台抽象阶段同时审视 channel contract，确保 runtime 不再隐式依赖 Discord 特性。

至少要明确这些能力接口：

- message send
- thread addressing / thread creation
- message edit/update
- attachment upload
- interactive prompt/button delivery
- task status signaling
- typing/progress hinting where supported

### 3. Reliability hardening

`0.8` 需要把剩余的可靠性短板收口：

- graceful shutdown contract
  - stop gateway cleanly
  - stop runtime workers cleanly
  - interrupt active subprocesses predictably
  - close SQLite/WAL cleanly
  - preserve restartable state
- startup config validation
  - schema-level validation of required fields/types
  - consistent fail-fast behavior in local and Docker modes
  - CLI binary validation available outside Docker too
- upgrade/migration contract
  - SQLite schema migrations
  - config compatibility changes
  - skill/workspace/runtime path migrations
  - explicit backup/rollback expectations
- message/delivery reliability
  - markdown-aware chunking
  - deterministic attachment/path fallback
  - defined rate-limit/request-queue behavior
- automation contract stabilization
  - current missed-job policy remains `skip`
  - make it explicit in docs and tests
  - test restart/reload behavior and hot-reload preservation
- concurrent isolation
  - multiple active threads/tasks remain isolated
  - concurrency behavior is tested, not assumed
- log hygiene
  - log rotation
  - log-level configuration
  - structured logging conventions
- user-visible error contract
  - agent failures show readable user-safe messages, not raw tracebacks
  - internal errors log full context but expose only concise summaries to the user
  - timeout / crash / storage failure each have defined user-facing behavior

### 4. Deployment hardening

`1.0` 是 self-hosted product，所以部署质量本身是 release bar。

`0.8` 应补齐：

- first-class `docker-compose`
- local 与 Docker 的安装/运行路径文档统一
- runtime directories / backup / restore instructions
- operator-facing restart and upgrade SOP
- health-check expectations for long-running service mode

## v0.9 — Release Candidate / Contract Freeze

`0.9` 不再引入大的能力面，只做契约冻结与验证。

目标：

- finish remaining service-layer extraction
- eliminate remaining adapter-owned business logic
- add end-to-end restart/recovery tests for:
  - chat
  - explicit skill invocation
  - runtime tasks
  - HITL waits
  - auth waits
  - automations
- validate upgrades from older state layouts
- tighten docs into operator-grade product docs
- defer or remove experimental surfaces not ready for long-term support

## 1.0 Acceptance Criteria

`1.0` candidate 必须满足：

1. fresh install 在本地 `venv` 和 Docker/compose 下都可用
2. direct chat、explicit skill invocation、`artifact`、`repo_change`、`skill_change` 在 Discord 上都可端到端运行
3. HITL approvals/checkpoints 和 auth-required pauses 能跨重启恢复
4. automations 能跨重启保留 runtime state，并遵循已文档化的 `skip` missed-job policy
5. `/doctor`、task logs、automation surfaces 足以定位故障，而不需要 source-diving
6. delivery 行为可预测：inline summary、attachment、local path fallback
7. graceful shutdown 不会留下不可恢复的 active state
8. config validation 在本地和 Docker 下都能早失败、清晰报错
9. version upgrades 在支持的版本跨度内有经过验证的 migration 行为
10. concurrent active threads/tasks 保持隔离并可正确恢复
11. Discord adapter 只负责 input parsing、output rendering、Discord SDK calls；task state transitions、HITL resolution、automation scheduling 等业务逻辑不再驻留在 adapter 中
12. `1.0` 不依赖 semantic retrieval、hybrid autonomy、guest mode、event-driven triggers 或第二 IM 平台

## Post-1.0 / 1.x

明确移出 `1.0` 关键路径：

- Slack
- Feishu/Lark
- WeChat
- semantic memory retrieval
- hybrid autonomy
- event-driven triggers
- guest session / tenant isolation
- free-text HITL
- remote object storage delivery
- richer autonomy / mission systems

平台扩展顺序在 `1.0` 之后再决定；`1.0` 前唯一要求是平台抽象做成 future-ready。

## Assumptions and Defaults

- `1.0` 继续是 **single-user / owner-operated**
- `1.0` 继续是 **self-hosted first**
- `1.0` 明确是 **Discord-only**
- 平台抽象在 `0.8` 做，是为了 **maintainability + future readiness**
- `automation runtime state`、`/doctor`、delivery abstraction、HITL baseline 都视为 **已完成基础**
- `0.8` 是 hardening + architecture release，不是 intelligence release
- `0.9` 是 RC / contract-freeze，不是新 feature wave
