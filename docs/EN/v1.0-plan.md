# v1.0 Plan: Stable Discord-First Self-Hosted Agent Runtime

## Summary

`1.0` should mean the product contract is stable, not that capability is maximized.

Given the current repo state, `1.0` should be defined as:

- **single-user / owner-operated**
- **self-hosted first**
- **generic runtime first**
- **reliability first**
- **Discord-only for the 1.0 platform scope**
- **future-ready platform abstraction**, but **no second IM platform is required for 1.0**

Version path reframed as:

- `0.8` = **1.0 hardening**
- `0.9` = **1.0 RC / contract freeze**
- `1.0` = **stable release**
- Slack / Feishu / WeChat, semantic retrieval, hybrid autonomy, guest mode, event-driven triggers, and remote object storage all move to **post-1.0 / 1.x**

## 1.0 Product Contract

`1.0` means one owner can install the system locally or with Docker, connect Discord, run Claude/Gemini/Codex, use skills and long tasks, complete approvals, recover after restart, and diagnose failures without source-diving.

Core `1.0` contract:

- Supported deployment:
  - local `venv`
  - Docker
  - `docker-compose` as the default operator path
- Supported agents:
  - Claude CLI
  - Gemini CLI
  - Codex CLI
- Supported platform:
  - Discord only
- Supported execution modes:
  - direct chat
  - explicit skill invocation
  - `artifact`
  - `repo_change`
  - `skill_change`
  - file-driven automations
- Supported HITL:
  - structured single-choice only
  - owner-gated approvals / confirmations / checkpoints
- Supported delivery:
  - inline summary
  - attachment when possible
  - local path fallback
- Supported persistence/recovery:
  - sessions
  - runtime tasks
  - HITL prompts
  - auth waits
  - automation runtime state
- Supported operator surfaces:
  - task status / logs
  - automation status / reload / enable / disable
  - auth status / login / clear
  - `/doctor`

Non-goals for `1.0`:

- multi-user or team tenancy
- guest mode
- second IM platform as a release gate
- event-driven triggers
- free-text HITL
- remote object storage delivery
- semantic retrieval / vector memory
- hybrid autonomy
- domain-specific skill packs as core release criteria

## v0.8 — 1.0 Hardening

### 1. Platform abstraction

The goal is not to ship a second platform for `1.0`, but to extract business logic from the current Discord adapter to reduce maintenance risk and prepare a clean entry point for future Feishu/Slack adapters.

Architecture choice:

- **service-layer architecture**
- platform adapters are responsible only for:
  - input parsing
  - output rendering
  - platform SDK calls
- shared service layer owns:
  - business logic
  - state transitions
  - result objects

Extraction order locked as:

1. **task control service**
   - highest value, most commands, heaviest state logic
2. **ask service**
   - core entry path, must decouple from platform early
3. **doctor service**
   - small scope, easy early win
4. **automation service**
   - medium scope, operator-facing importance
5. **auth service**
   - can be deferred
6. **memory/operator service**
   - `memories/forget/memorize` can be extracted last

Requirements:

- not required to extract every command in one pass
- `0.8` completion bar is:
  - the heaviest business logic no longer resides in the platform adapter
  - adding a second platform does not require copying Discord-specific business branches

### 2. BaseChannel contract review

Alongside the abstraction pass, review and extend the channel interface so the core runtime is not secretly Discord-specific.

The `1.0`-ready channel contract should explicitly cover:

- message send
- thread addressing / thread creation
- message edit/update
- attachment upload
- interactive prompt/button delivery
- task status signaling
- typing/progress hinting where supported

### 3. Reliability hardening

`0.8` should close the remaining product-grade reliability gaps:

- graceful shutdown contract
  - stop gateway cleanly
  - stop runtime workers cleanly
  - interrupt active subprocesses predictably
  - close SQLite/WAL cleanly
  - preserve restartable state
- startup config validation
  - schema-level validation of required fields/types
  - consistent fail-fast behavior in local and Docker modes
  - CLI binary validation available outside Docker too
- upgrade/migration contract
  - SQLite schema migrations
  - config compatibility changes
  - skill/workspace/runtime path migrations
  - explicit backup/rollback expectations
- message/delivery reliability
  - markdown-aware chunking
  - deterministic attachment/path fallback
  - defined rate-limit/request-queue behavior
- automation contract stabilization
  - current missed-job policy remains `skip`
  - make it explicit in docs and tests
  - test restart/reload behavior and hot-reload preservation
- concurrent isolation
  - multiple active threads/tasks remain isolated
  - concurrency behavior is tested, not assumed
- log hygiene
  - log rotation
  - log-level configuration
  - structured logging conventions
- user-visible error contract
  - agent failures show readable user-safe messages, not raw tracebacks
  - internal errors log full context but expose only concise summaries to the user
  - timeout / crash / storage failure each have defined user-facing behavior

### 4. Deployment hardening

`1.0` is a self-hosted product, so deployment quality is part of the release bar.

`0.8` should add:

- first-class `docker-compose`
- documented parity between local and Docker installs
- clear runtime directories, backup paths, and restore instructions
- operator-facing restart and upgrade SOPs
- health-check expectations for long-running service mode

## v0.9 — Release Candidate / Contract Freeze

`0.9` should not introduce major new surfaces. It should freeze and verify the `1.0` contract.

Primary goals:

- finish remaining service-layer extraction
- eliminate remaining adapter-owned business logic
- add end-to-end restart/recovery tests for:
  - chat
  - explicit skill invocation
  - runtime tasks
  - HITL waits
  - auth waits
  - automations
- validate upgrades from older state layouts
- tighten docs into operator-grade product docs
- defer or remove experimental surfaces not ready for long-term support

## 1.0 Acceptance Criteria

A `1.0` candidate must satisfy all of these:

1. Fresh install works in local `venv` mode and Docker/compose mode.
2. Direct chat, explicit skill invocation, `artifact`, `repo_change`, and `skill_change` all work end-to-end on Discord.
3. HITL approvals/checkpoints and auth-required pauses survive restart and resume correctly.
4. Automations survive restart, preserve runtime state, and follow the documented `skip` missed-job policy.
5. `/doctor`, task logs, and automation surfaces are sufficient to diagnose a broken run without source-diving.
6. Delivery is deterministic: inline summary, attachment when possible, local path fallback otherwise.
7. Graceful shutdown leaves the system restartable without corrupting active state.
8. Config validation fails early and clearly in both local and Docker modes.
9. Version upgrades across supported releases have tested migration behavior.
10. Concurrent active threads/tasks remain isolated and recover correctly.
11. Discord adapter is limited to input parsing, output rendering, and Discord SDK calls; task state transitions, HITL resolution, and automation scheduling no longer reside in the adapter.
12. No `1.0` acceptance path depends on semantic retrieval, hybrid autonomy, guest mode, event-driven triggers, or a second IM platform.

## Post-1.0 / 1.x

These move off the `1.0` critical path:

- Slack
- Feishu/Lark
- WeChat
- semantic memory retrieval
- hybrid autonomy
- event-driven triggers
- guest session / tenant isolation
- free-text HITL
- remote object storage delivery
- richer autonomy / mission systems

Platform expansion order should remain open until post-1.0 usage context is clearer. The only pre-1.0 requirement is that the service-layer architecture and channel contract make a second platform materially easier.

## Assumptions and Defaults

- `1.0` remains **single-user / owner-operated**
- `1.0` remains **self-hosted first**
- `1.0` is explicitly **Discord-only**
- platform abstraction is done in `0.8` for **maintainability + future readiness**
- `automation runtime state`, `/doctor`, delivery abstraction, and HITL baseline are treated as **already implemented foundations**
- `0.8` is a hardening + architecture release, not an intelligence release
- `0.9` is an RC / contract-freeze cycle, not another feature wave
